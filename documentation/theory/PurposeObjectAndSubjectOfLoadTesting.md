# Purpose, Object, and Subject of Load Testing

## Overview
Load testing is the process of evaluating the performance of an application or system under different load conditions. It helps assess how efficiently the software operates under real or extreme scenarios, identifying bottlenecks and limitations.

In standards terms, load testing measures the **performance efficiency** quality characteristic. ISO/IEC 25010:2023 defines performance efficiency as the *"capability of a product to perform its functions within specified time and throughput parameters and be efficient in the use of resources under specified conditions"* and splits it into three sub-characteristics:

- **Time behaviour** — response times and throughput rates meet requirements.
- **Resource utilization** — the product uses no more than the specified amount of resources (CPU, memory, storage, network) while performing its functions.
- **Capacity** — the maximum limits of a product parameter (concurrent users, transaction throughput, database size, bandwidth) meet requirements.

Load testing is a *type of performance testing*. ISTQB defines **performance testing** as *"an umbrella term including any kind of testing focused on performance (responsiveness) of the system or component under different volumes of load"*, and **load testing** specifically as testing that *"focuses on the ability of a system to handle increasing levels of anticipated realistic loads resulting from transaction requests generated by controlled numbers of concurrent users or processes."*

The goal of load testing is to ensure that the system can:
- Handle the expected request volumes (**anticipated, realistic** load — not just theoretical peaks).
- Maintain stability and correctness (functional correctness must hold *under* load, not only at rest).
- Meet SLA (Service Level Agreement) and non-functional requirements, expressed as measurable SLOs/SLIs (see *Load Testing Metrics*).

## Theory: Key Concepts

### Load Testing Purpose
Main tasks of load testing include:
- **Performance validation:** determine how many requests per second (RPS) or transactions per second (TPS) the system can handle while still meeting its response-time targets.
- **Scalability validation:** assess how the system responds to load increase and resource scaling — i.e. the ability to grow (more users, more data) *without violating* the currently specified performance requirements (ISTQB *scalability testing*).
- **Bottleneck identification:** detect the single component that limits end-to-end performance (database, application server, connection pool, external API, network). Under load, the slowest resource — the *bottleneck* — caps total throughput regardless of headroom elsewhere.
- **Production readiness:** model expected and peak loads before release, so capacity decisions are data-driven rather than guessed.
- **Resilience validation:** test at or beyond the limits (ISTQB *stress testing*) to find breaking points and to observe **recovery** behaviour after the overload subsides.
- **Regression protection:** confirm that new changes do not silently degrade performance versus a known baseline.

Business value: performance defects found in production are the most expensive to fix and the most visible to users; load testing shifts that discovery left, converting an outage risk into a pre-release engineering task.

### Load Testing Object
An object is any part of the system under load. Typical objects:
- **REST API:** the core of microservice architecture and, in this project, the primary object under test.
- **Microservices:** individual services running in the system, tested both in isolation and end-to-end.
- **Databases:** query handling, locking, connection-pool behaviour, and index efficiency under high load.
- **Infrastructure:** servers, load balancers, reverse proxies, networks, container CPU/memory limits.
- **Message queues / event pipelines:** asynchronous processing systems (Kafka, RabbitMQ) where throughput and backlog matter.

**Key metrics for objects:**
- **Latency / response time:** time to process a request. Note that some authoritative sources (e.g. Kleppmann, *Designing Data-Intensive Applications*) distinguish the two — *response time* is what the client observes (service time + network + queueing delay), whereas *latency* is the duration a request spends waiting to be handled. Many APM tools use the terms interchangeably; state your definition explicitly.
- **Throughput:** amount of work processed per unit of time (requests/s, transactions/s, or bytes/s). This is the "Traffic" golden signal in Google SRE terms — a measure of demand placed on the system.
- **Error rate:** percentage of failed requests. "Failure" must be defined explicitly (e.g. HTTP 4xx/5xx, timeouts, or business-level assertion failures); the criterion is tool-configurable, not universal.
- **Resource utilization:** CPU, memory, network, and disk usage of the object while serving the load (maps directly to ISO 25010 *resource utilization*).

### Load Testing Subject
Subjects are participants and consumers of test results:
- **End users:** API consumers whose perceived experience the whole exercise ultimately protects.
- **Developers:** optimize code and queries based on test results and profiling.
- **QA / performance engineers:** design, execute, and analyze the tests; own the non-functional coverage.
- **DevOps / SRE engineers:** manage infrastructure, monitoring, capacity, and tuning; own SLIs/SLOs and the error budget.
- **Product owners / managers:** need system performance insights to make release and capacity decisions.

## Detailed Overview of Key Aspects

### Types of Load Testing
This course groups scenarios into a practical taxonomy (Peak / Steady / Stress / Volume — see *Load Testing Theory*). For alignment with the industry standard, the ISTQB **Performance Testing** syllabus enumerates these named types:

- **Load testing** — handling increasing levels of *anticipated realistic* loads from controlled numbers of concurrent users/processes.
- **Stress testing** — behaviour *at or beyond* the limits of anticipated workloads, including reduced-resource conditions.
- **Scalability testing** — ability to meet *future* efficiency requirements as users/data grow, without violating current requirements.
- **Spike testing** — ability to respond correctly to sudden bursts of peak load and *return afterwards to a steady state*.
- **Endurance (soak) testing** — stability over a long time frame; surfaces resource issues such as memory leaks, unclosed DB connections, or exhausted thread pools. ("Soak" is the common industry synonym; ISTQB's term is *endurance testing*.)
- **Concurrency testing** — impact of actions occurring *simultaneously* (e.g. many users logging in at once); race conditions are notoriously hard to reproduce.
- **Capacity testing** — how many users/transactions the system supports while still meeting its performance objectives.

> Note: *Volume testing* and *configuration testing* are widely used in practice but are **not** enumerated as types in ISTQB's Performance Testing syllabus (they appear in older/advanced material). Treat the course's "Volume" block as an established industry practice rather than a term from that syllabus.

### Load Testing Metrics
- **Latency / response time:**
    - Average (arithmetic mean) response time — useful but *misleading on its own*, because latency distributions are right-skewed and the mean hides the slow tail (Google SRE, *Monitoring Distributed Systems*).
    - **Time to First Byte (TTFB):** time from the start of the request to the first byte of the response. This is a standardized, measurable quantity via the W3C Navigation/Resource Timing `responseStart` attribute (MDN).
    - **Time to Last Byte (TTLB):** time until the last byte of the response is received (≈ TTFB + body-transfer time). Note: TTLB is a vendor/industry term, **not** defined by any W3C/IETF standard.
    - **Percentiles (p50 / p90 / p95 / p99):** the value below which that share of observations falls. `p50` is the median (the typical case); `p90/p95/p99` describe the tail (plausible worst case). Prefer percentiles over the average for latency SLOs. Caveat: different tools compute percentiles with slightly different estimation methods (nearest-rank vs interpolation), so values can differ marginally.
- **Throughput:**
    - Requests per second (RPS) and/or transactions per second (TPS). **Convention:** one business *transaction* may comprise multiple HTTP *requests* (e.g. a checkout = several calls), so TPS ≤ RPS. This RPS-vs-TPS split is an industry convention, not a formal standard.
    - Data volume transferred per second (bytes/s).
- **Error rate:**
    - Percentage of 4xx/5xx HTTP codes, request timeouts, and connection errors (e.g. exhausted database connection pool). Always state which conditions count as "failed".
- **Resource utilization:** CPU (%), memory, disk I/O, network bandwidth — collected from the system under test, not the load generator.
- **Connection pool usage:** pool utilization and connection failures due to an exhausted pool — a classic hidden ceiling that the *closed* load model can expose.
- **SLA / SLO / SLI compliance** (Google SRE definitions):
    - **SLI** — *"a carefully defined quantitative measure of some aspect of the level of service"* (e.g. the proportion of requests faster than 300 ms).
    - **SLO** — *"a target value or range of values for a service level that is measured by an SLI"* (e.g. 99% of requests < 300 ms over 28 days).
    - **SLA** — *"an explicit or implicit contract with your users that includes consequences of meeting (or missing) the SLOs it contains."*
    - **Error budget** = `1 − SLO` (Google SRE Workbook) — the allowed amount of unreliability in a window; it turns "reliable enough" into an objective, spendable quantity.
- **Apdex score (Application Performance Index):** a 0–1 user-satisfaction score defined by the Apdex specification as
    `Apdex_T = (Satisfied + Tolerating / 2) / Total`,
    with three zones relative to a chosen threshold **T**: *Satisfied* (RT ≤ T), *Tolerating* (T < RT ≤ 4T), *Frustrated* (RT > 4T). Frustrated samples count only in the denominator. The optional qualitative bands are Excellent (0.94–1.00), Good (0.85–0.93), Fair (0.70–0.84), Poor (0.50–0.69), Unacceptable/UNAX (0.00–0.49).
- **Peak load, success rate, latency distribution (histogram), retries, queue time, dropped requests, load efficiency** — supporting indicators used to explain *why* the primary metrics behave as they do.

> **Tail latency and fan-out.** Tail percentiles matter disproportionately in distributed systems. Dean & Barroso ("The Tail at Scale", CACM 2013) show that if one server has a p99 of 1 s and a user request must gather responses from 100 such servers in parallel, then ≈63% of user requests exceed 1 s (1 − 0.99¹⁰⁰). Optimizing the average would not fix this; only the tail matters.

### Why Load Testing is Critical for REST API
REST APIs often handle high concurrent request volumes and are the backbone of microservices. Without load testing:
- Bottlenecks lead to delays that cascade across dependent services.
- Resource overuse (CPU, memory, connections) causes crashes or restarts under real traffic.
- Behaviour becomes unpredictable as user count grows, and tail latency degrades faster than the average.

## Technology Stack
- **JMeter (and K6/Gatling):** load scenario creation and load generation.
- **Prometheus/Grafana:** metrics scraping, storage, and dashboards for the system under test.
- **InfluxDB:** time-series store for JMeter's own result stream (via the Backend Listener).
- **Docker:** test-environment orchestration and per-container resource limits (so the object under test has a known, reproducible resource envelope).

## Example REST API Testing Scenario
1. **Preparation:**
    - Define SLOs and their SLIs (e.g. p95 response time ≤ 200 ms; error rate < 1%).
    - Choose target API endpoints and prepare realistic test data (users, seeded records).
2. **Configuration:**
    - Create test plans in JMeter or K6; choose the correct workload model (open vs closed — see *Load Testing Theory*).
    - Define load levels (normal, peak, extreme) and, for a max-throughput search, a stepped ramp.
3. **Execution:**
    - Run tests with real-time monitoring of both the load generator and the system under test.
4. **Analysis:**
    - Collect metrics (percentile latency, throughput, error rate, resource usage); identify the bottleneck.
5. **Recommendations:**
    - Optimize the bottleneck, re-test under identical conditions, and compare against the baseline.
